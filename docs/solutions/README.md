## 机制说明

### 目的

集中管理开发和使用过程中遇到的问题及其解决方案，便于：
- 追踪问题根因
- 复用解决方案
- 知识沉淀
- 快速检索

### 目录命名规范

| 命名方式 | 示例 | 说明 |
|---------|------|------|
| **kebab-case** | `at-mention` | 短横线连接的小写单词 |
| **简洁明确** | `context-loss` | 一眼看出问题类型 |
| **英文优先** | `multi-channel` | 便于国际化和检索 |

### 文档结构

每个问题目录包含：

```
solutions/[issue-name]/
├── README.md          # 问题总览（可选）
├── specific.md        # 特定场景方案
├── general.md         # 通用/根本解决方案
├── analysis.md        # 深度分析（可选）
└── test-cases.md      # 测试验证（可选）
```

### 新增问题流程

1. **识别问题**: 在开发/使用中发现问题
2. **创建目录**: `mkdir solutions/issue-name`
3. **撰写文档**: 至少包含 `specific.md` 和 `general.md`
4. **更新索引**: 在本 README 中添加条目
5. **提交 GitHub**: `git add docs/solutions/ && git commit`
