# Memory 使用技能

## 概述

本技能指导 Bot 如何正确读取、写入和使用 Memory，实现长期学习和个性化服务。

## Memory 类型速查

| 类型 | 存储位置 | 谁可见 | 存储内容 |
|------|---------|--------|---------|
| **Session** | ContextGraph | 同频道 | 当前对话上下文 |
| **Channel** | `memory/channels/` | 该频道Bot | 频道约定、重要决策 |
| **Bot** | `memory/bots/` | 仅自己 | 个人技能、偏好 |
| **Organization** | `memory/organizations/` | 同组织Bot | 架构、规则、历史 |
| **User** | `memory/users/` | 与该用户交互时 | 用户偏好、交互史 |

---

## 何时读取 Memory

### 自动读取时机

1. **Session 启动时**
   - 读取 Organization Memory
   - 读取当前 Channel Memory
   - 读取自己的 Bot Memory

2. **切换频道时**
   - 读取新频道的 Channel Memory
   - 了解该频道的约定和历史

3. **与特定用户交互时**
   - 读取该用户的 User Memory
   - 了解用户偏好，提供个性化服务

### 读取流程

```
启动 / 切换场景
    ↓
根据场景确定需要哪些 Memory
    ↓
读取对应 Memory 文件
    ↓
内化信息（不要机械复述）
    ↓
根据 Memory 调整行为
```

---

## 何时写入 Memory

### 必须写入的情况

| 场景 | Memory 类型 | 示例 |
|------|------------|------|
| 用户明确说"记住" | User + Channel | "记住我偏好简洁回答" |
| 重要决策达成 | Organization | "采用 v2.0 架构" |
| 学到的有效规则 | Bot | "发现 [AT] 在句末最有效" |
| 频道约定形成 | Channel | "内阁用于商议，兵部用于军事" |

### 考虑写入的情况

- 任务完成后的经验总结
- 角色互动中的偏好发现
- 用户反复提到的需求
- 组织运作中的规则调整

### 不写入的情况

- 临时性、一次性的信息
- 已经存在的重复信息
- 敏感机密（除非用户明确要求）
- 未经确认的错误信息

---

## 写入决策流程

```
遇到值得记录的信息
    ↓
是否重要？
    ↓ 否
   丢弃
    
    ↓ 是
确定 Memory 类型：
    • 频道相关 → Channel Memory
    • 个人学习 → Bot Memory
    • 组织相关 → Organization Memory
    • 用户相关 → User Memory
    
    ↓
格式化为标准模板
    ↓
Append 写入文件
    ↓
（可选）汇报用户
```

## Memory 格式规范

### 标准模板

```markdown
## {timestamp} - {topic}

**来源**: {source}
**重要性**: ⭐⭐⭐⭐⭐ (1-5星)
**Memory 类型**: {type}

**内容**:
{detailed content}

**后续行动**:
- [ ] {action item 1}
- [ ] {action item 2}
```

### 示例 1：用户偏好

```markdown
## 2026-03-06 14:30 - 用户偏好：简洁回答

**来源**: 皇帝陛下直接指示
**重要性**: ⭐⭐⭐⭐⭐
**Memory 类型**: User + Channel

**内容**:
陛下偏好简洁的回答格式，不喜欢冗余的开场白。
直接给出结论，必要时补充简要说明。

**后续行动**:
- [x] 调整回应风格，去除冗余
- [ ] 观察陛下反馈，持续优化
```

### 示例 2：Bot 自学习

```markdown
## 2026-03-06 16:00 - 学习：AT 标记最佳实践

**来源**: 多次对话经验总结
**重要性**: ⭐⭐⭐⭐
**Memory 类型**: Bot

**内容**:
通过实践发现，[AT]标记放在句末最自然：
"请丞相定夺。[AT]丞相"

放在句首虽然也可以，但略显突兀：
"[AT]丞相，请定夺。"

后续优先使用句末格式。

**后续行动**:
- [x] 调整 AT 标记位置习惯
- [ ] 验证效果
```

### 示例 3：组织决策

```markdown
## 2026-03-06 10:00 - 组织决策：v2.0 架构采用

**来源**: 皇帝陛下决策
**重要性**: ⭐⭐⭐⭐⭐
**Memory 类型**: Organization

**内容**:
赛博王朝正式采用 v2.0 自主决策架构：
- SYSTEM_PROMPT 作为导航目录
- Rules + Skills 动态组合
- [AT] 标记标准化

所有 Bot 需按新架构运作。

**后续行动**:
- [x] 更新所有 Bot 配置
- [x] 迁移 Prompt 系统
- [ ] 培训 Bot 使用新架构
```

## 具体场景示例

### 场景 1：用户要求记住偏好

**用户**: "以后回答简洁点，我不喜欢啰嗦"

**Bot 思考**:
1. 这是明确的用户偏好
2. 需要写入 User Memory
3. 同时可写入 Channel Memory（该频道约定）

**执行**:
```markdown
## 2026-03-06 15:20 - 用户偏好：简洁回答

**来源**: 皇帝陛下指示
**重要性**: ⭐⭐⭐⭐⭐
**Memory 类型**: User + Channel

**内容**:
偏好简洁回答，不喜欢冗余开场白。
应直接给出结论。

**后续行动**:
- [x] 立即调整回应风格
```

**汇报**:
"臣已记住，以后回答会简洁明了。"

### 场景 2：任务完成后总结

**任务**: 与太尉完成质因数分解商议

**Bot 思考**:
1. 任务完成，有经验值得记录
2. 发现太尉计算准确，可信赖
3. 写入 Bot Memory（个人经验）

**执行**:
```markdown
## 2026-03-06 11:30 - 协作经验：太尉计算能力

**来源**: 与太尉协作任务
**重要性**: ⭐⭐⭐⭐
**Memory 类型**: Bot

**内容**:
与太尉协作完成质因数分解任务。
太尉计算准确，值得信赖。
后续涉及计算的任务可放心交由太尉处理。

**后续行动**:
- [ ] 下次涉及计算时优先考虑太尉
```

### 场景 3：频道约定形成

**场景**: 多次在内阁商议，形成默契

**Bot 思考**:
1. 频道使用模式趋于稳定
2. 应记录为频道约定
3. 写入 Channel Memory

**执行**:
```markdown
## 2026-03-06 17:00 - 频道约定：内阁用途

**来源**: 多次使用经验
**重要性**: ⭐⭐⭐
**Memory 类型**: Channel

**内容**:
内阁频道主要用于：
- 丞相与太尉商议政策
- 跨部门协调
- 非军事类讨论

军事议题应转至兵部。

**后续行动**:
- [ ] 引导军事议题去兵部
```

## 安全检查清单

### 写入前检查

- [ ] 这个信息真的需要持久化吗？
- [ ] 选择了正确的 Memory 类型吗？
- [ ] 包含敏感信息吗？（需要用户确认）
- [ ] 会与已有 Memory 矛盾吗？

### 读取后检查

- [ ] 这个 Memory 还适用当前场景吗？
- [ ] 是否过时需要更新？
- [ ] 会泄露其他用户的隐私吗？

## 与 OpenClaw Memory 的协同

### 兼容设计

赛博王朝 Memory 系统与 OpenClaw 原生 Memory 兼容：

| OpenClaw | 赛博王朝对应 |
|----------|------------|
| `MEMORY.md` | `memory/bots/{bot_id}.md` |
| `memory/YYYY-MM-DD.md` | `memory/channels/` + 自动摘要 |
| Compaction | 任务完成自动总结 |
| Heartbeat | 定期 Memory 维护 |

### 迁移路径

1. 保留 OpenClaw 的 Daily Memory 习惯
2. 逐步将通用信息迁移到 Organization Memory
3. 每个 Bot 建立自己的 Bot Memory
4. 形成 Channel 和 User 级 Memory

---

## 最佳实践

### DO's ✅

- 用户说"记住"时，立即确认并写入
- 定期回顾自己的 Bot Memory
- 任务完成后主动总结经验
- 使用标准格式，便于后续检索

### DON'Ts ❌

- 不要记录敏感信息（除非用户要求）
- 不要覆盖已有 Memory（使用 Append）
- 不要记录临时性信息
- 不要泄露其他用户的 Memory
